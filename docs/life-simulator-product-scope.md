# 人生模拟器产品开发范围

## 目的

本文固定现实人生模拟器从 v3 验证程序走向可发布产品时的长期模块边界、依赖方向和交付闸门。

动态任务状态、负责人、优先级、阻塞、排期和 Project Status 由 Linear 维护。本文不替代 Linear，也不记录阶段百分比或当前执行状态。

## 产品原则

1. `clock.totalWeeks` 继续作为唯一权威时间；年度选择推进 52 周，恋爱选择推进 1 周。
2. Engine、内容、View Model、UI、视觉资产和工具链保持分离。
3. 所有状态变化继续通过正式 Command 和原子结算完成。
4. 相同 seed、policy、content 和正式身份输入必须可确定性重放。
5. UI 只提交正式 `choiceId`，不得在页面脚本中直接修改 Engine 状态或自行随机。
6. AI 生成内容只能作为候选草稿，必须通过自动校验、Simulation 预览和人工审核后才能进入正式内容。
7. 正式资源由 `realmruckus/world` 自己维护，不复制 Classic 主题资产，不依赖外部热链。

## 产品模块

### P9–P13：内容与人生系统

负责扩充年度事件、恋爱完整路径、经济与人生目标、NPC 家庭与社会关系网络，以及 Derived Metrics、评分和结局分布校准。

内容层必须覆盖主要年龄阶段，并验证事件重复率、关系路径、经济输入、NPC 引用完整性和结局可达性。

### P15：内容 Schema 与审核工具

负责校验 Event、Choice、Command、Relationship Rule、Metric、Ending、Save 和 Asset 引用。

必须检查重复 ID、悬空引用、非法状态转换、未知 Command 参数、不可达事件、不可达结局和 Metric 依赖循环。

### P18：开局身份构筑与个人资料卡

玩家开局通过卡牌选择：

- 性别；
- 星座；
- 家庭背景；
- 父亲职业；
- 母亲职业。

家庭背景是现实人生模拟器的起源层，影响初始财富、教育资源、家庭支持、社会连接、健康环境、事件条件和事件权重。

父母职业必须进入正式数据模型，不得仅作为展示文字。开局确认后生成持续更新、可放大查看的个人资料卡。

### P16：UI 与人机交互架构

建立稳定的全屏游戏画布，并明确人物状态、当前事件、选择区、关系、资产、档案、成就和系统工具的信息层级。

年度、恋爱、危机、关系定性和结局必须具有清晰、稳定且可恢复的交互状态。移动端必须支持 Safe Area、Safari 工具栏避让、长文本内部滚动和至少 44×44 CSS px 的主要触控区域。

### P17：人物、场景与事件视觉资产

建立人物年龄阶段、家庭、学校、职场、住宅、医院、城市、旅行、恋爱、危机、成就、档案和结局的视觉资产系统。

所有正式资源必须具有唯一 Asset ID、版本、来源、许可和审核状态。Event、Ending、Achievement 与 UI 只引用 Asset ID，不在 Engine 或组件中硬编码文件路径。

### P19：真实卡牌交互、展开与换牌

所有玩家选择统一使用 `LifeChoiceCardViewModel` 表达，包括身份、教育、大学专业、职业、关系、行动、危机和结局。

卡牌交互必须支持：

- 手牌扇形、横向牌列或堆叠收纳；
- 桌面端悬停或键盘聚焦展开；
- 移动端点击预览、再次确认或拖出打出；
- 长按或详情按钮进入全屏放大；
- 完整标题、插画、条件、效果、风险、来源和禁用原因；
- 拿起、归位、打出、弃置、翻面和换牌；
- 在大学专业、职业和其他适用阶段执行“全部换牌”。

换牌次数、成本、冷却和结果必须由正式 Deck/Offer Model 决定，写入 Timeline，可保存并可确定性重放。

卡牌可采用成熟数字卡牌游戏的通用交互范式，但不得复制任何现有商业游戏的商标、卡框、材质、字体、声音、动画或视觉识别。

### P20：卡牌场景演出与视觉反馈

表现层采用以下状态机：

```text
Draw → Inspect → Commit → Resolve → Reveal → Apply → Archive → Return
```

演出不得改变 Engine 结果、随机序列或存档时序。动画中断或页面刷新后必须从正式存档恢复，不得重复结算。

关键结果必须同时提供文字反馈；Reduced Motion 和低性能降级模式必须保留完整信息与操作能力。

### P21：内容编辑器与模拟预览工作台

建立浏览器端内容编辑、Schema 校验、差异预览、固定 seed 重放和批量 Simulation 预览能力。

编辑器覆盖年度事件、恋爱事件、危机事件、家庭模板、父母职业、教育专业、职业、NPC、Metric、Ending 和 Asset 映射。

草稿与正式内容必须严格分离。未经批准的内容不得写入正式目录或生产部署。

### P22：AI 内容生产与审核流水线

根据家庭背景、年龄阶段、教育、职业、经济和关系条件生成结构化内容候选，包括事件、NPC、职业路径、结局、测试 Fixture、插画 Prompt 和 Asset Manifest 草稿。

流水线必须记录模型、Prompt 版本、输入参数、内容哈希、来源和审核状态，并执行：

```text
生成草稿 → Schema 校验 → 去重与一致性检查 → Simulation 预览 → 人工审核 → 批准入库
```

未经批准的 AI 内容不得进入正式发布路径。

### P14：浏览器与移动端最终验收

最终验收覆盖新人生、身份构筑、年度选择、恋爱、危机、关系定性、结局、归档、成就、本地存档、刷新恢复、JSON 导入导出、清除数据、卡牌展开、全屏详情、换牌、动画中断恢复和 Reduced Motion。

Node 测试、浏览器自动验收和实际移动端回读均为独立闸门，任何一项失败都构成阻塞。

## 依赖方向

```text
P9–P13 内容与人生系统
        ↓
P15 内容 Schema 与审核
        ↓
P18 开局身份构筑与资料卡
        ↓
P16 UI 与人机交互架构
        ↓
P17 图片与视觉资产
        ↓
P19 真实卡牌交互
        ↓
P20 场景演出与视觉反馈
        ↓
P21 内容编辑器
        ↓
P22 AI 内容生产流水线
        ↓
P14 浏览器与移动端最终验收
```

P21 和 P22 可以在核心产品交互稳定后并行扩展，但所有正式输出仍必须经过 P15 校验，并由 P14 覆盖最终发布路径。

## 交付阶段

### 可试玩版本

至少完成：

- 基础年龄段事件与重复控制；
- 开局身份构筑；
- 个人资料卡；
- 稳定全屏 UI；
- 统一卡牌选择、展开与详情查看；
- 基础换牌；
- Node 回归和核心浏览器验收。

### 正式 MVP

在可试玩版本基础上增加：

- 完整恋爱、经济、NPC 和结局分布；
- 正式人物与场景资产；
- 卡牌演出与 Reduced Motion；
- 内容编辑器；
- AI 内容审核流水线；
- 完整浏览器、iPhone 和存档迁移验收。

## 时间估算基线

在一名熟悉项目的开发者配合 AI、依赖顺序稳定且审核及时的条件下：

- 可试玩版本：约 5–6 周；
- 正式 MVP：约 10–14 周。

该估算是产品范围基线，不代表当前排期承诺。实际日期、负责人和任务状态由 Linear 维护。
