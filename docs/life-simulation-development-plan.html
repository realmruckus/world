M00 · Life Simulation Engine · Plan v3.1

现实人生模拟引擎开发计划 v3.1

本项目先以现实世界单页单机人生重开游戏验证玩法，再沉淀为 WORLD 可复用的角色、事件、关系、档案与 Simulation 引擎。无数据库，内容使用 JSON，状态保存在浏览器本地。

一、验证目标

  • 人生按年、恋爱按周的双节奏是否好玩。
  • 事件、Command、延迟后果、关系与结局能否完全数据驱动。
  • 档案、成就和多周目是否形成持续游玩动力。
  • 相同 seed、policy、content 和版本是否可重放。
  • Engine 是否可迁移到 WORLD NPC 与世界事件系统。

二、核心循环

创建人物 → 年度事件 → 选择 → Command 原子结算 → +52周
                   ↓
              进入恋爱阶段
                   ↓
恋爱周事件 → 选择 → 关系维度变化 → +1周 → 关系定性
                   ↓
结局 → 九维评分 → 成就 → 档案 → 重开

三、系统架构

UI / SVG Renderer
       ↓
Life Controller
       ↓
Event Selector / Command Executor / Relationship Engine
       ↓
Canonical Clock / Metric Engine / Ending Engine / Archive
       ↓
JSON Content + localStorage v3
       ↓
Simulation Runner

四、正式数据来源

  • Contract v3.1：结算顺序与不变量。
  • Event Schema v3.1：事件与 Command。
  • Save Schema v3.1：模块化人物与存档。
  • Relationship Rules v3：状态转换 intent 与阈值。
  • Metric Registry v3：指标公式。
  • Metric DSL v3：语法、函数、默认值与错误策略。
  • Ending Rules v3：结局条件与评分。

五、时间与中断

  • totalWeeks 是唯一权威时间。
  • 年度选择 +52 周；恋爱选择 +1 周。
  • 危机为临时插入，不改变当前 stage；按事件自身 timeScale 推进后继续原阶段。
  • 冷却与 scheduled 全部保存绝对总周数。

六、关系玩法

八项关系维度：attraction、love、trust、conflict、dependence、respect、passion、commitment。状态转换只能通过 Relationship Rules 中登记的 from、to、intent 和阈值完成。

  • 单段目标 6–16 次选择。
  • 软上限 26 周，硬上限 52 周。
  • 空白周自动快进。
  • 非法转换拒绝整次 Command，禁止自动跳过承诺节点。

七、事件内容

事件由 Scene、Narration、Choice、Commands、Future Commands、Probability Modifiers 和 AI Tags 构成。

  • 年度人生事件：60–80。
  • 恋爱周事件：36–48。
  • 恋爱链式节点：12–18。
  • 危机中断事件：8–12。
  • 结局前事件：6–8。
  • 首轮总事件不少于 120。

八、结局与评分

Ending 只读取 achievement、relationship、health、money、legacy、freedom、knowledge、contribution、adventure。Metric 公式使用受限声明式 DSL，禁止 eval；未知函数、未知指标、循环依赖和无默认值的缺失路径直接失败。

九、开发阶段

  1. P0：冻结 Contract v3.1、Schemas v3.1、Relationship Rules、Metric Registry、Metric DSL 与 Ending Rules。
  2. P1：测试优先实现 canonical clock、Command Executor、Schema/不变量校验和 v2→v3 迁移。
  3. P2：实现模块化人物、关系八维、Metric DSL Parser 与 Relationship Engine。
  4. P3：迁移年度事件，完成年度垂直切片。
  5. P4：实现恋爱周阶段、危机插入和关系定性。
  6. P5：实现九维 Metric 与 Ending Engine。
  7. P6：实现本地存档、导入导出、档案与成就。
  8. P7：实现 runLife 与 runBatch，执行至少 10,000 次批量模拟。
  9. P8：移动端 UI、程序化 SVG 和三轮实际游玩审核。

十、P1 测试矩阵

  • 年度 +52 周、恋爱 +1 周、51→52 跨年。
  • 危机插入不改变 stage，按自身 timeScale 推进。
  • 冷却与 scheduled 截止周准确。
  • 所有 Command 参数严格校验，非法操作整次回滚。
  • stageStartedAtWeeks 与关系时间戳不晚于 totalWeeks。
  • v2→v3 存档迁移补齐最低状态字段。

十一、P2–P5 测试矩阵

  • 关系维度截断、intent 阈值、非法跳级和硬上限定性。
  • Metric DSL 不执行 JavaScript，未知函数和循环依赖报错。
  • 空关系、空经历与零资产按登记默认值计算。
  • Ending 只读取九个高层 Metric。
  • 相同 seed、policy、content 和版本得到相同结果。

十二、质量门槛

  • 单次人生 8–15 分钟。
  • 同事件单生重复不超过 2 次；连续同类不超过 3 次。
  • 伪选择率低于 15%。
  • 恋爱卡死率为 0。
  • 10,000 次 Simulation 中无不可解释异常；单一主结局原则上不超过 35%。

十三、完成标准

Engine、单页 Web、JSON 内容、SVG、自动测试、10,000 人生 Simulation、本地存档、档案、成就、移动端审核和 Linear 同步全部完成；实现与所有 v3.1 正式输入一致。
