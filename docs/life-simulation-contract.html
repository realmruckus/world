RR-95 · Contract Freeze v3

Life Simulation Engine Contract v3

RR-95 正式定位为现实人生模拟引擎的前置验证器。单页单机产品用于验证通用时间、事件、Command、关系、派生指标、结局、档案与批量 Simulation，并为 WORLD 的角色与事件系统提供可复用实现。

一、权威时间

  • clock.totalWeeks 是唯一持久化时间。
  • ageYears=floor(totalWeeks/52)，weekOfYear=totalWeeks%52，不得独立写入。
  • 年度事件选择成功后增加 52 周；恋爱事件选择成功后增加 1 周。
  • 进入、退出阶段和查看界面不推进时间。
  • scheduled 与 cooldown 均保存绝对截止周。

二、阶段与中断

  • stage=life 时普通事件按年推进；stage=romance 时普通恋爱事件按周推进。
  • 危机使用临时插入模型，不创建阶段栈，不改变当前 stage，不改变 stageStartedAtWeeks。
  • 中断事件必须声明自己的 timeScale；按事件声明推进，结束后继续原阶段。
  • 恋爱硬上限以 totalWeeks-stageStartedAtWeeks 计算，中断所消耗时间计入阶段长度。
  • 死亡与不可继续状态优先于任何阶段事件。

三、关系引擎

关系保存八项 0–100 维度：attraction、love、trust、conflict、dependence、respect、passion、commitment。事件 Command 主要修改维度；状态由 Relationship Engine 根据阈值、前置状态和明确承诺事件判定。

none → potential → dating → exclusive → cohabiting → engaged → married
                    ↘ paused → broken_up → no_contact
  • 状态不得仅因单次数值越界自动跳过必要承诺节点。
  • 明确承诺、同居、订婚、结婚与分手事件可携带状态转换意图，Engine 必须复核阈值。
  • 单段恋爱目标 6–16 个玩家选择，软上限 26 周，硬上限 52 周。
  • 硬上限必须触发关系定性事件；空白周自动快进。

四、Command Effect

所有事件结果均为白名单 Command List。首版 Command：

AddStat / SetStat / AddResource / SetResource
AddTag / RemoveTag / SetFlag / ClearFlag
CreateRelationship / ModifyRelationship / RequestRelationshipTransition / EndRelationship
AppendExperience / ScheduleEvent / UnlockAchievement
SetCareer / SetEducation / EnterStage / ExitStage

每种 Command 必须按 op 进行参数校验。任一 Command 非法时整次选择回滚。

五、事件对象

事件至少包含 Scene、Narration、Choice、Immediate Commands、Future Commands、Probability Modifiers 与 AI Tags。事件内容不得嵌入 JavaScript 或任意表达式。

  • Scene：程序化 SVG 场景标识与文字说明。
  • Narration：标题、描述与结果文本。
  • Choice：1–4 个选择。
  • Commands：即时与延迟 Command。
  • Probability Modifiers：基于标签、属性、关系和历史的声明式权重修正。
  • AI Tags：领域、语气、风险、生命周期和内容审核标签。

六、冷却与计划事件

  • 事件内容写 {amount,unit}。
  • 选择结算时立即换算：year×52，week×1。
  • 存档只保存 cooldowns[eventId]=cooldownUntilTotalWeeks。
  • scheduled 只保存 dueAtTotalWeeks。
  • 可触发条件为当前 totalWeeks 大于等于截止周。

七、派生指标

机器可读唯一来源为 data/life-derived-metrics.json。统一流程：

Input → Normalize → Weight → Sum → Round → Clamp 0–100
  • 复杂指标必须由纯函数计算。
  • 加载存档后重新计算，缓存值不得作为权威来源。
  • 未知指标必须报错，不得默认为 0。
  • 公式注册表定义输入、归一化、权重、空数据与截断语义。

八、结局解耦

Ending Rules 只读取九个高层 Metric：achievement、relationship、health、money、legacy、freedom、knowledge、contribution、adventure，以及 ageYears、Tag 和 Flag。不得直接读取职业、朋友数量或具体人物字段。

结局选择：死亡原因 → 条件筛选 → priority → 加权分数 → id 稳定排序。只产生一个主结局和最多六个次级标签。

九、人物模块

Identity / Body / Mind / Career / Finance
Relationships / Health / Inventory / Location / History

M00 可只实现当前玩法需要的子集，但 LifeState 的模块边界必须保持，避免 UI、事件和 Engine 直接依赖扁平字段。

十、Simulation 接口

runLife(seed, policy, content) → LifeArchive
runBatch(count, seedStart, policy, content) → SimulationReport

报告至少统计结局分布、事件覆盖率、事件重复率、职业与关系路径、成就解锁率、平均操作数、恋爱卡死率和无效选择率。相同输入必须可重放。

十一、不变量与验证

  • stageStartedAtWeeks ≤ totalWeeks。
  • 所有关系时间戳不得晚于 totalWeeks。
  • 所有属性与关系维度截断至 0–100。
  • 所有 Metric 名必须存在于注册表。
  • 中断事件不得隐式改变 stage。
  • 选择、Command、时间推进、派生计算和保存必须原子完成。

十二、冻结状态

Contract v3 已冻结。修改权威时间、冷却模型、中断语义、关系维度、Ending Metric 接口、Command 语义或存档 Key 均属于破坏性变更。
