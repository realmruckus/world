Life Simulation Engine Contract v3.1

本契约冻结现实人生模拟引擎的时间、事件、Command、关系、派生指标、结局、存档与 Simulation 接口。单页单机产品负责验证其可玩性，并为 WORLD 的角色与事件系统提供可复用实现。

一、权威时间

  • clock.totalWeeks 是唯一持久化时间。
  • ageYears=floor(totalWeeks/52)，weekOfYear=totalWeeks%52，不得独立写入。
  • 年度事件选择成功后增加 52 周；恋爱事件选择成功后增加 1 周。
  • 进入、退出阶段和查看界面不推进时间。
  • scheduled 与 cooldown 均保存绝对截止周。

二、阶段与中断

  • stage=life 时普通事件按年推进；stage=romance 时普通恋爱事件按周推进。
  • 危机使用临时插入模型，不创建阶段栈，不改变当前 stage 或 stageStartedAtWeeks。
  • 中断事件按自身 timeScale 推进，结束后继续原阶段。
  • 恋爱硬上限以 totalWeeks-stageStartedAtWeeks 计算，中断消耗的时间计入阶段长度。
  • 死亡与不可继续状态优先于任何阶段事件。

三、关系引擎

关系保存 attraction、love、trust、conflict、dependence、respect、passion、commitment 八项 0–100 维度。正式转换规则唯一来源为 data/life-relationship-rules.json。

  • 状态转换必须包含明确 intent，并满足 from、to、维度阈值和冲突上限。
  • 不得自动跳过承诺节点；未知状态或非法转换直接拒绝整次 Command。
  • 单段目标 6–16 个玩家选择，软上限 26 周，硬上限 52 周。
  • 硬上限必须触发允许的关系定性 intent；空白周自动快进。

四、Command Effect

事件结果均为白名单 Command List：AddStat、SetStat、AddResource、SetResource、AddTag、RemoveTag、SetFlag、ClearFlag、CreateRelationship、ModifyRelationship、RequestRelationshipTransition、EndRelationship、AppendExperience、ScheduleEvent、UnlockAchievement、SetCareer、SetEducation、EnterStage、ExitStage。

每种 Command 必须按 op 进行参数校验。任一 Command 非法时，Command、时间推进、派生计算与保存全部回滚。

五、事件对象

事件至少包含 Scene、Narration、Choice、Immediate Commands、Future Commands、Probability Modifiers 与 AI Tags。事件内容不得嵌入 JavaScript 或任意可执行代码。

六、冷却与计划事件

  • 事件内容使用 {amount,unit} 描述持续时间。
  • 选择结算时立即换算：year×52，week×1。
  • 存档只保存 cooldowns[eventId]=cooldownUntilTotalWeeks 整数。
  • scheduled 只保存 dueAtTotalWeeks。
  • 当前 totalWeeks 大于等于截止周时可再次触发。

七、派生指标

指标定义唯一来源为 data/life-derived-metrics.json；公式语言、允许函数、空数据和默认值唯一来源为 data/life-metric-dsl.json。

  • 公式按依赖拓扑顺序计算，循环依赖、未知函数、未知指标和无默认值的缺失路径均报错。
  • 公式只允许声明式 DSL，不使用 eval、Function 或 JavaScript 解释执行。
  • 统一执行 Normalize → Weight → Sum → Round → Clamp 0–100。
  • 加载存档后重新计算，缓存值不得作为权威来源。

八、结局解耦

Ending Rules 只读取 achievement、relationship、health、money、legacy、freedom、knowledge、contribution、adventure 九个高层 Metric，以及 ageYears、Tag 和 Flag。不得直接读取职业、朋友数量或具体人物字段。

选择顺序：死亡原因 → 条件筛选 → priority → 加权分数 → id 稳定排序。只产生一个主结局和最多六个次级标签。

九、人物最低状态

LifeState 保持 Identity、Body、Mind、Career、Finance、Relationships、Health、Inventory、Location、History 模块边界。Metric Engine 所需最低字段必须存在：

identity.childrenCount
body.fitness
mind.smarts / mind.empathy / mind.stress
career.level / career.freeTime
finance.assets / finance.debt / finance.businessAssets / finance.reputation
health.health

十、Simulation 接口

runLife(seed, policy, content) → LifeArchive
runBatch(count, seedStart, policy, content) → SimulationReport

报告至少统计结局分布、事件覆盖率、重复率、职业与关系路径、成就解锁率、平均操作数、恋爱卡死率和无效选择率。相同 seed、policy、content 与版本必须可重放。

十一、不变量

  • stageStartedAtWeeks ≤ totalWeeks。
  • 所有关系时间戳不得晚于 totalWeeks。
  • 所有属性与关系维度截断至 0–100。
  • 所有 Metric 名与函数必须存在于注册表。
  • 中断事件不得隐式改变 stage。
  • 存档 Key 为 realmruckus.world.life.v3，schemaVersion=3。

十二、冻结状态

Contract v3.1 已冻结。修改权威时间、冷却模型、中断语义、关系维度与转换规则、Metric DSL、Ending Metric 接口、Command 语义或存档 Key 均属于破坏性变更。
