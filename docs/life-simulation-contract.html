人生模拟数据与规则契约 v2

本页冻结 M00 第二版实现契约。人生主线按年展开，恋爱阶段按周展开；玩家每次完成一个选择，只推进当前阶段的一个时间单位。

一、正式文件

  • life-event.schema.json

    事件、时间尺度、条件、选择、Effect、延迟后果和链式事件的数据约束。

  • life-save.schema.json

    混合时间轴、当前人生、历史人生、成就、设置和存档版本的数据约束。

  • life-ending-rules.json

    死亡原因、主结局、次级标签、优先级和评分权重。

二、混合时间模型

人生阶段

stage = life，timeScale = year。每次选择后年龄推进 1 年，并进入下一年度事件。

恋爱阶段

stage = romance，timeScale = week。每次选择后推进 1 周，用于约会、关系磨合、表白、同居、冲突和承诺等连续事件。

Life Clock
├─ ageYears：完整年龄
├─ weekOfYear：当前年龄内第 0–51 周
├─ totalWeeks：从出生起累计周数
├─ stage：life | romance
├─ timeScale：year | week
└─ stageWeek：当前恋爱阶段累计周数
  • 人生阶段选择：ageYears += 1，totalWeeks += 52，weekOfYear 保持 0。
  • 恋爱阶段选择：totalWeeks += 1，weekOfYear += 1；达到 52 周时年龄增加 1，周数归零。
  • 进入恋爱阶段时不额外推进时间；退出恋爱阶段时也不额外推进时间。
  • 所有时间推进只能发生在选择成功结算之后，查看档案、切换面板和导出存档不推进时间。
  • 恋爱阶段结束后回到人生阶段；下一次人生事件选择仍推进 1 年。
  • 普通非恋爱事件不得在恋爱周阶段混入随机池；公共危机或健康紧急事件需要显式标记可中断恋爱阶段。

三、人物状态结构

LifeState
├─ id / seed / rngCursor / createdAt
├─ identity
│  ├─ name / genderExpression / birthYear / region
│  ├─ familyBackground / educationId / careerId
│  ├─ maritalStatus / childrenCount / housingStatus
├─ clock
│  ├─ ageYears / weekOfYear / totalWeeks
│  └─ stage / timeScale / stageWeek
├─ alive
├─ stats
│  ├─ health / happiness / smarts / charm
│  ├─ fitness / discipline / empathy / stress
├─ resources
│  ├─ cash / income / debt / assets
│  ├─ careerLevel / reputation / freeTime
│  └─ financialSecurity
├─ relationships[]
├─ tags[] / flags{}
├─ experiences[] / timeline[]
├─ cooldowns{}
├─ scheduled[]
├─ pendingEvent
└─ ending

所有核心属性使用 0–100。现金、收入、债务和资产使用非负整数；净资产由资产与债务派生，不作为第二份正式字段存储。关系必须使用独立对象，不再使用汇总数字代替真实人物。

四、事件时间约束

  • 每个事件必须声明 timeScale。
  • 恋爱类事件必须声明 type = romance、stage = romance、timeScale = week。
  • 人生主线事件默认声明 stage = life、timeScale = year。
  • 冷却、延迟后果和计划事件必须同时保存数值与单位，不得再使用含义不明的单一整数。
  • 所有计划事件统一换算为 dueAtTotalWeeks 存储，避免跨年和跨阶段时计算歧义。
  • 恋爱事件可使用 weekInStage 条件控制第几周进入表白、确认关系、同居或分手节点。

五、Effect 操作表

数值

add_stat、set_stat、add_resource、set_resource。属性结算后统一截断到合法范围。

标签与状态

add_tag、remove_tag、set_flag、clear_flag。

身份与经历

set_identity、set_career、set_education、append_experience。

关系

create_relationship、modify_relationship、end_relationship。

阶段与未来事件

enter_stage、exit_stage、schedule_event。进入或退出阶段本身不推进时间。

禁止做法

禁止在事件 JSON 中嵌入 JavaScript、任意表达式或直接修改未知路径；所有行为必须由白名单操作解释执行。

六、事件选择顺序

  1. 根据 clock 确认当前 stage 与 timeScale。
  2. 结算当前 totalWeeks 已到期的 scheduled 项。
  3. 检查当前阶段的强制链式事件和里程碑事件。
  4. 检查允许中断当前阶段的危机事件。
  5. 筛选时间尺度、年龄、阶段周数、标签、Flag、属性、资源、职业、教育和关系条件。
  6. 排除冷却中和 oncePerLife 已完成事件。
  7. 按 priority 选择最高优先层。
  8. 仅在同一优先层中按 weight 加权抽取。
  9. 使用 seed 与 rngCursor 产生可重放随机结果。

七、选择结算顺序

验证 choiceId
→ 复制当前状态
→ 依序执行即时 effects
→ 写入 futureEffects / nextEventId
→ 更新 cooldown 与 oncePerLife 记录
→ 写入 experience
→ 写入不可变 timeline 条目和 delta 摘要
→ 按当前 timeScale 推进 1 年或 1 周
→ 重新计算派生值
→ 检查结构合法性
→ 检查生命结束与结局
→ 保存本地存档

任一 Effect 或时间推进非法时，整次选择不得产生部分写入。第一版浏览器实现使用内存复制实现原子性。

八、恋爱阶段边界

  • 恋爱阶段不是整个人生的永久周制，而是一个可进入、可退出的局部高密度事件阶段。
  • 进入条件通常是建立潜在伴侣关系、接受约会或触发明确恋爱机会。
  • 退出条件包括确认长期关系、结婚、分手、失联、主动结束或达到阶段最长周数。
  • 原型默认单次恋爱阶段最多 104 周；达到上限仍未产生明确结果时，必须触发关系定性事件。
  • 每周只出现一个主要选择，避免同一周同时处理多个恋爱事件。
  • 婚后日常、育儿和长期家庭生活回归按年推进；只有明确的恋爱或关系危机短阶段可再次切入周制。

九、结局判定

  1. 先确定生命是否结束以及 deathCauseId。
  2. 筛选满足 conditions 的主结局。
  3. 按 priority 从高到低比较。
  4. 同优先级按该结局 score 公式比较。
  5. 仍相同时按 id 字典序确定，确保可重放。
  6. 只保留一个 primaryEnding。
  7. 额外计算最多六个 secondaryTags。
  8. 人生总评分按 wellbeing、relationships、growth、stability、legacy 五维加权生成。

十、本地存档

由于时间模型属于破坏性变更，localStorage 正式 Key 提升为 realmruckus.world.life.v2，根对象 schemaVersion = 2。

  • 存档必须保存完整 clock、seed 与 rngCursor。
  • 历史人生最多保存 100 份；超出后删除最旧档案。
  • 导入前必须进行 Schema 校验，失败时不得覆盖现有存档。
  • v1 迁移为 v2 时：旧 age 转为 ageYears，weekOfYear 与 stageWeek 设为 0，totalWeeks 设为 age × 52，stage 设为 life。
  • 旧冷却年数转换为 { remaining, unit: "year" }；旧 dueAge 转为 dueAtTotalWeeks = dueAge × 52。

十一、冻结状态

Contract v2 已冻结。本次变更替代 Contract v1 的单一年度时间模型。后续删除字段、改变时间推进语义、改变阶段边界或改变存档 Key，必须再次提升 schemaVersion 并提供迁移测试。
