人生模拟数据与规则契约 v1

本页冻结 M00 第一版实现契约。后续 Engine、事件内容、界面、测试与存档迁移必须以本契约和对应 JSON Schema 为准，不再继续沿用早期原型中的扁平字段。

一、正式文件

  • life-event.schema.json

    事件、条件、选择、Effect、延迟后果和链式事件的数据约束。

  • life-save.schema.json

    当前人生、历史人生、成就、设置和存档版本的数据约束。

  • life-ending-rules.json

    死亡原因、主结局、次级标签、优先级和评分权重。

二、人物状态结构

LifeState
├─ id / seed / rngCursor / createdAt
├─ identity
│  ├─ name / genderExpression / birthYear / region
│  ├─ familyBackground / educationId / careerId
│  ├─ maritalStatus / childrenCount / housingStatus
├─ age / alive
├─ stats
│  ├─ health / happiness / smarts / charm
│  ├─ fitness / discipline / empathy / stress
├─ resources
│  ├─ cash / income / debt / assets
│  ├─ careerLevel / reputation / freeTime
│  └─ financialSecurity
├─ relationships[]
├─ tags[] / flags{}
├─ experiences[] / timeline[]
├─ cooldowns{}
├─ scheduled[]
├─ pendingEvent
└─ ending

所有核心属性使用 0–100。现金、收入、债务和资产使用非负整数；净资产由资产与债务派生，不作为第二份正式字段存储。关系必须使用独立对象，不再使用 family/friends/partner 三个汇总数字代替真实人物。

三、Effect 操作表

数值

add_stat、set_stat、add_resource、set_resource。属性结算后统一截断到合法范围。

标签与状态

add_tag、remove_tag、set_flag、clear_flag。标签用于可查询分类，Flag 用于唯一链路和明确状态。

身份与经历

set_identity、set_career、set_education、append_experience。

关系

create_relationship、modify_relationship、end_relationship。所有修改必须指向 relationshipId 或明确角色选择规则。

未来事件

schedule_event 将事件或 Effect 写入 scheduled 队列，使用 dueAge 结算，不依赖真实时间。

禁止做法

禁止在事件 JSON 中嵌入 JavaScript、函数名、任意表达式或直接修改未知路径；所有行为必须由白名单操作解释执行。

四、事件选择顺序

  1. 结算当前年龄到期的 scheduled 项。
  2. 检查强制链式事件和里程碑事件。
  3. 检查危机保底事件。
  4. 筛选年龄、标签、Flag、属性、资源、职业、教育和关系条件。
  5. 排除冷却中和 oncePerLife 已完成事件。
  6. 按 priority 选择最高优先层。
  7. 仅在同一优先层中按 weight 加权抽取。
  8. 使用 seed 与 rngCursor 产生可重放随机结果。

事件数据必须明确区分 priority 与 weight：priority 决定先处理哪一层事件，weight 只决定同层候选之间的概率。

五、选择结算顺序

验证 choiceId
→ 复制当前状态
→ 依序执行即时 effects
→ 写入 futureEffects / nextEventId
→ 更新 cooldown 与 oncePerLife 记录
→ 写入 experience
→ 写入不可变 timeline 条目和 delta 摘要
→ 重新计算派生值
→ 检查结构合法性
→ 检查生命结束与结局
→ 保存本地存档

任一 Effect 非法时整次选择不得产生部分写入。第一版浏览器实现使用内存复制实现原子性。

六、结局判定

  1. 先确定生命是否结束以及 deathCauseId。
  2. 筛选满足 conditions 的主结局。
  3. 按 priority 从高到低比较。
  4. 同优先级按该结局 score 公式比较。
  5. 仍相同时按 id 字典序确定，确保可重放。
  6. 只保留一个 primaryEnding。
  7. 额外计算最多六个 secondaryTags。
  8. 人生总评分按 wellbeing、relationships、growth、stability、legacy 五维加权生成。

隐藏结局仅通过明确 Flag 解锁，不允许仅依靠极低随机概率覆盖其他结局。

七、本地存档

localStorage 正式 Key 为 realmruckus.world.life.v1。根对象必须包含 schemaVersion、appVersion、savedAt、currentLife、archives、achievements 和 settings。

  • 每次选择结算、创建新人生、结束人生和修改设置后保存。
  • 历史人生最多保存 100 份；超出后删除最旧档案。
  • 导入前必须进行 Schema 校验，失败时不得覆盖现有存档。
  • schemaVersion 变化必须提供显式迁移函数；无法迁移时保留原数据并提示导出。
  • 存档保存 seed 与 rngCursor，确保下一次事件抽取可继续重放。

八、与早期原型的迁移

现有 js/life-sim-engine.js 仅是探索代码。下一阶段重构时：

  • money、salary 迁入 resources；
  • education、career 迁入 identity；
  • 扁平 relationships 汇总值替换为关系对象数组；
  • 旧 effect 对象替换为白名单 effects 数组；
  • 旧 condition 替换为统一 conditions；
  • 旧 ending 内联对象替换为 endingId 和正式规则表；
  • 随机数调用全部经过 seed 与 rngCursor。

九、冻结状态

Contract v1 已冻结。后续允许增加可选字段和新枚举值，但删除字段、改变字段语义、改变结算顺序或改变存档 Key 均属于破坏性变更，必须提升 schemaVersion 并提供迁移测试。
