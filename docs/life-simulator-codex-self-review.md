# 人生模拟器 Codex 自我审核与迭代标准

## 目的

本文定义 Codex 在提交人工审核前必须执行的自我审核、问题分类、修复迭代和证据交付标准。

Codex 自我审核是人工审核的前置闸门，不替代 Linear 中的正式人工审核 Issue，也不能自行批准内容、UI、视觉、权重或发布。

## 总体流程

```text
实现完成
→ 冻结当前 Head
→ 独立自我审核
→ 发现问题并分类
→ 新增失败测试
→ 最少修复
→ 完整回归
→ 再次自我审核
→ 满足退出条件
→ 提交人工审核
```

至少执行两轮：

1. 第一轮：实现者视角，检查需求覆盖、测试和范围；
2. 第二轮：审查者视角，假设实现是错误的，主动寻找反例、缺口和范围外修改。

不得以“测试通过”代替第二轮审查。

## 审核对象

Codex 必须审核：

- Linear Issue、阻塞关系和验收条件；
- Base 与 Head 之间的完整 Diff；
- 每个 Commit 的单一目标；
- 新增与修改的代码、Schema、Fixture、数据、测试和 CI；
- 测试结果、Simulation 报告、浏览器报告或资源报告；
- 未执行项和人工保留项；
- GitHub 与 Linear 是否一致。

## 审核方法

### 1. 需求追踪矩阵

为每项验收建立对应关系：

| 验收项 | 实现文件 | 测试文件 | 验证命令 | 结果 | 证据 |
|---|---|---|---|---|---|

每个验收项必须至少对应一个可重复验证证据。无法自动验证的内容必须标记为“需要人工审核”，不得自行判定通过。

### 2. Diff 审核

逐文件检查 Base...Head Diff：

- 是否存在范围外文件；
- 是否修改正式规则、权重、内容或 UI；
- 是否出现重复实现、临时代码、调试输出或无用文件；
- 是否把 draft/fixture 写入正式目录；
- 是否修改或弱化既有测试；
- 是否存在隐式随机、时间依赖或不可重放行为；
- 是否存在硬编码路径、ID 或未经批准的默认值。

### 3. 反例驱动审核

对每个新增 Contract、状态机或 Validator，至少构造：

- 一个正常输入；
- 一个最小边界输入；
- 一个缺失字段输入；
- 一个额外字段输入；
- 一个非法引用输入；
- 一个重复 ID 输入；
- 一个非法状态转换输入；
- 一个不支持版本输入；
- 一个确定性重放输入。

只要任何反例未被现有测试覆盖，必须先增加失败测试再修复。

### 4. 双实现一致性审核

如果同一契约同时存在 JSON Schema、手写 Validator、View Model 或 Adapter，必须验证：

- 相同有效输入全部接受；
- 相同无效输入全部拒绝；
- 错误不会被静默修复；
- 不返回部分有效对象；
- 版本与状态约束一致。

发现不一致即为阻塞。

### 5. 确定性审核

相同 seed、输入、policy、content 和版本必须产生相同：

- Loader 结果；
- Offer/Deck 结果；
- Simulation 报告；
- Timeline；
- Ending 与 Metric；
- 序列化输出。

至少重复运行两次并比较机器可读结果。不得只比较控制台摘要。

### 6. 回归审核

必须运行：

- 当前 Issue 新增测试；
- 当前模块全部测试；
- 全部 `tests/life-*.test.mjs`；
- `npm test`；
- JavaScript 语法检查；
- JSON、Schema 或资源校验；
- Issue 要求的 Simulation、浏览器或性能检查；
- GitHub Actions 对应命令。

任何既有测试失败都构成阻塞。禁止删除、跳过、改名或弱化测试来获得通过。

### 7. 数据与内容边界审核

必须确认：

- 候选内容只位于 `draft` 或 `fixtures`；
- `approved` 状态不能由 Codex 自行写入；
- 不把社会假设、家庭价值、星座解释、视觉偏好或结局意义写成正式规则；
- 示例值有明确 fixture 标记；
- Fixture 不被正式 Loader 当成生产内容自动加载。

### 8. 安全与失败关闭审核

检查：

- 非法输入是否被拒绝而非自动补全；
- 未知 ID、未知 Command、未知 Metric 是否失败关闭；
- 数值是否拒绝 `NaN`、`Infinity`、负数和越界值；
- 页面刷新、重复点击或动画中断是否会重复结算；
- 未批准内容是否无法进入部署路径。

### 9. 性能与规模审核

适用时检查：

- 10,000 人生 Simulation 是否真实执行；
- 报告是否包含执行数量、失败数量、seed、版本和错误摘要；
- 重复运行结果是否一致；
- 无明显内存、时间或文件体积异常；
- 性能结论有真实测量，不使用主观描述。

### 10. 提交与同步审核

检查：

- 每个 Commit 只完成一个目标；
- Commit Message 类型正确；
- 没有空 Commit 或无关格式化；
- PR 保持 Draft，未自动合并；
- Linear 记录 Commit、Blob、文件、测试、CI、未执行项和人工审核入口；
- GitHub 与 Linear 状态一致。

## 问题等级

### 阻塞

任一情况出现即不得提交人工审核：

- 违反正式规则或仓库职责；
- 失败关闭缺失；
- 确定性失败；
- 既有测试失败；
- 未批准内容进入正式目录；
- Simulation、CI 或必要验证未执行；
- 数据丢失、重复结算或版本不兼容；
- 明显偏离 Linear 验收范围。

### 必修

不一定破坏核心运行，但会使人工审核无法有效进行：

- 缺少关键边界测试；
- Fixture 代表性不足；
- 报告字段不完整；
- 错误信息不可定位；
- Schema、Validator 或文档不一致；
- PR 证据不完整。

### 建议

不影响当前验收，可以进入后续独立任务：

- 非必要重构；
- 可选性能优化；
- 更丰富的 Fixture；
- 非当前范围的 UI、美术或内容改进。

Codex 不得为了处理“建议”扩大当前 Issue 范围。

## 自我迭代规则

每轮必须：

1. 冻结并记录当前 Head；
2. 生成审核清单；
3. 列出阻塞、必修、建议；
4. 对阻塞和必修项先新增失败测试；
5. 提交独立修复 Commit；
6. 运行完整回归；
7. 重新从 Base...Head 审核，不只看最后一个 Commit；
8. 更新问题状态和证据。

最多连续执行三轮自我修复。三轮后仍有阻塞，必须停止并在 Linear 明确记录，不得继续无边界迭代。

## 自我审核退出条件

只有全部满足，才能提交人工审核：

- 所有验收项有追踪证据；
- 阻塞项为 0；
- 必修项为 0；
- 建议项已记录且未扩大范围；
- 新增测试、完整回归和 CI 全部通过；
- 必要 Simulation 或浏览器验证已真实执行；
- 关键文件已回读；
- PR Diff 无范围外修改；
- 候选内容仍与正式目录隔离；
- PR 保持 Draft；
- Linear 已同步；
- 明确列出仍需人工判断的事项。
